Si une entreprise veut créer un système Retrieval-Augmented Generation pour ses documents RH (contrats, congés, politiques internes, CV, fiches de paie, procédures…), l’ingénieur IA suit généralement ces étapes :

1. Analyse du besoin métier

Avant le développement :

Comprendre les problèmes actuels
recherche lente
documents dispersés
réponses RH répétitives
manque de centralisation
Identifier les utilisateurs
employés
RH
managers
administration
Définir les cas d’usage
“Combien de jours de congé reste-t-il ?”
“Quelle est la procédure de télétravail ?”
“Trouver le contrat de X”
“Résumer les règles RH”
2. Audit et collecte des données

L’ingénieur analyse les sources :

PDF
DOCX
Excel
Emails
Base de données RH
SharePoint
Drive interne
ERP / SAP / Odoo

Puis il vérifie :

qualité des données
doublons
documents obsolètes
langues utilisées
confidentialité
3. Définir l’architecture globale

Architecture classique :

Documents RH
      ↓
Ingestion Pipeline
      ↓
Chunking + Embedding
      ↓
Vector Database
      ↓
Retriever
      ↓
LLM
      ↓
Chat RH / API
4. Choisir les technologies
LLM

Exemples :

OpenAI GPT
Anthropic Claude
Meta Llama
Mistral AI Mistral
Vector Database
Pinecone
Qdrant
Weaviate
Chroma
PostgreSQL + pgvector
Frameworks
LangChain
LlamaIndex
FastAPI
React
5. Création du pipeline d’ingestion

L’ingénieur développe :

Extraction de texte

Depuis :

PDF
DOCX
OCR pour scans
Nettoyage
suppression des caractères inutiles
correction encodage
normalisation
Chunking

Découpage des documents en petits morceaux.

Exemple :

chunk size = 500 tokens
overlap = 50
6. Génération des embeddings

Chaque chunk est transformé en vecteur numérique grâce à un modèle embedding.

Exemple :

OpenAI embeddings
BGE
E5
Sentence Transformers

Le but :

retrouver les documents similaires sémantiquement.
7. Stockage dans la base vectorielle

Chaque chunk stocke :

texte
embedding
source
date
auteur
département RH
niveau de confidentialité
8. Développement du Retriever

Le retriever :

reçoit la question
cherche les chunks pertinents
retourne le top K résultats

Exemple :

Question → recherche vectorielle → top 5 chunks
9. Connexion avec le LLM

Le LLM reçoit :

question utilisateur
contexte retrouvé

Puis génère une réponse basée sur les documents RH.

10. Ajout de sécurité (TRÈS important)

Pour les RH :

Gestion des permissions

Exemple :

un employé ne voit que ses données
RH voit tous les dossiers
Authentification
SSO
OAuth
LDAP
Active Directory
Chiffrement
documents
base vectorielle
API
Logs & audit

Tracer :

qui consulte quoi
quelles réponses sont générées
11. Création de l’interface

Possibilités :

Chat RH

Type :
“ChatGPT interne”

Dashboard admin
upload documents
monitoring
statistiques
coût LLM
Recherche avancée
filtres
département
date
type document
12. Évaluation du système

Tests :

précision des réponses
hallucinations
groundedness
temps de réponse
recall@K

L’ingénieur peut créer :

benchmark automatique
dataset QA RH
13. Optimisation avancée

Ensuite viennent les améliorations :

Hybrid Search
vectorielle + keyword
Re-ranking

Améliore les résultats.

Auto Chunking

Choix intelligent des chunks.

Multi-agent RAG

Agents spécialisés :

congés
paie
recrutement
Graph RAG

Relier :

employés
contrats
départements
14. Déploiement

Infrastructure :

Docker
Kubernetes
Cloud ou on-premise

CI/CD :

GitHub Actions
GitLab CI

Monitoring :

logs
coût
latence
erreurs
15. Maintenance continue

L’ingénieur doit :

mettre à jour les documents
réindexer
surveiller qualité
gérer nouveaux modèles LLM
optimiser les coûts
